358 /* Note that all access to mApplying are made by NetworkOPsImp,
359 and must be done under that class's lock. This avoids the overhead of
360 taking a separate lock, and the consequences of a race condition are
361 nearly-zero.
362
363 1. If there is a race condition, and getApplying returns false when it
364 should be true, the transaction will be processed again. Not that
365 big a deal if it's a rare one-off. Most of the time, it'll get
366 tefALREADY or tefPAST_SEQ.
367 2. On the flip side, if it returns true, when it should be false, then
368 the transaction must have been attempted recently, so no big deal if
369 it doesn't immediately get tried right away.
370 3. If there's a race between setApplying and clearApplying, and the
371 flag ends up set, then a batch is about to try to process the
372 transaction and will call clearApplying later. If it ends up
373 cleared, then it might get attempted again later as is the case with
374 item 1.
375 */
376 bool mApplying = false;
