195// This is a special transaction used for creating accounts through a
196// cross-chain transfer. A normal cross-chain transfer requires a "chain claim
197// id" (which requires an existing account on the destination chain). One
198// purpose of the "chain claim id" is to prevent transaction replay. For this
199// transaction, we use a different mechanism: the accounts must be claimed on
200// the destination chain in the same order that the `XChainCreateAccountCommit`
201// transactions occurred on the source chain.
202//
203// This transaction can only be used for XRP to XRP bridges.
204//
205// IMPORTANT: This transaction should only be enabled if the witness
206// attestations will be reliably delivered to the destination chain. If the
207// signatures are not delivered (for example, the chain relies on user wallets
208// to collect signatures) then account creation would be blocked for all
209// transactions that happened after the one waiting on attestations. This could
210// be used maliciously. To disable this transaction on XRP to XRP bridges, the
211// bridge's `MinAccountCreateAmount` should not be present.
212//
213// Note: If this account already exists, the XRP is transferred to the existing
214// account. However, note that unlike the `XChainCommit` transaction, there is
215// no error handling mechanism. If the claim transaction fails, there is no
216// mechanism for refunds. The funds are permanently lost. This transaction
217// should still only be used for account creation.
